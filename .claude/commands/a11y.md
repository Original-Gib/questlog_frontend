Perform an accessibility audit of the QuestLog frontend.

If a specific file or component is provided in the arguments, audit only that. Otherwise audit all files under `src/components/` and `src/views/`.

Arguments: $ARGUMENTS

## What to check

**Semantic HTML**
- Headings follow a logical hierarchy (`h1` → `h2` → `h3`); no levels are skipped; only one `h1` per page
- Interactive elements use native elements (`<button>`, `<a>`, `<input>`) rather than divs/spans with click handlers
- `<a>` tags are used for navigation, `<button>` for actions
- Lists use `<ul>`/`<ol>`/`<li>` where appropriate
- Landmark elements (`<main>`, `<nav>`, `<footer>`, `<header>`) are used meaningfully

**Labels and names**
- Every `<input>`, `<select>`, and `<textarea>` has an associated `<label>` (via `for`/`id`) or `aria-label`
- Every icon-only button has `aria-label`
- Images have meaningful `alt` text; decorative images have `alt=""`

**ARIA**
- Loading states use `:aria-busy="true"` on the element being updated
- Error messages use `role="alert"` so screen readers announce them automatically
- Inputs in an error state use `:aria-invalid="'true'"` and `aria-describedby` pointing to the error element
- Toggle buttons (e.g. theme toggle) use `role="switch"` with `:aria-checked`
- `aria-hidden="true"` is applied to decorative SVGs and icons

**Keyboard navigation**
- All interactive elements are reachable and operable via keyboard
- Focus styles are visible — check for `focus-visible:` ring classes; do not use `outline-none` without a replacement
- Modal dialogs or drawers (if any) trap focus correctly

**Colour and contrast**
- Text uses semantic token classes (`text-fg-dim`, `text-fg`, `text-muted`) rather than hardcoded colours
- Never rely on colour alone to convey information (e.g. error states also have text, not just a red border)

**Forms**
- Required fields are marked `required` (and ideally indicated visually)
- Form submission errors are announced via `role="alert"`
- `autocomplete` attributes are set on common fields (`email`, `new-password`, `current-password`)

## Output format

Group findings by file. For each issue include:
- The element or line causing the problem
- The specific WCAG criterion it relates to (e.g. WCAG 1.3.1, 4.1.2)
- A concrete fix recommendation

End with a summary: total issues found, breakdown by severity (blocker / warning / suggestion).
